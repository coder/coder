# AI Governance Audit Events

> [!NOTE]
> AI Governance features require the [AI Governance Add-On](./ai-governance.md).
> As of Coder v2.32, deployments without the add-on will not be able to access AI Gateway or Agent Firewall.

The AI Governance Add-On produces audit signals across three distinct subsystems,
each with its own storage and query surface:

| Subsystem                                                                                            | Storage                       | Access                                              |
|------------------------------------------------------------------------------------------------------|-------------------------------|-----------------------------------------------------|
| AI provider configuration, AI seat consumption, group AI budgets, chats, tasks, and personal secrets | `audit_logs` table            | [Audit Logs UI and API](#querying-audit-log-events) |
| AI Gateway interceptions, token usage, MCP calls, and model reasoning                                | `aibridge` session store      | [AI Gateway Sessions UI](./ai-gateway/audit.md)     |
| Agent Firewall enforcement decisions                                                                 | Workspace and connection logs | Workspace log viewer                                |

### Field tracking notes

Some fields are recorded as "changed but value withheld" rather than storing the actual value.
This prevents sensitive material from appearing in audit diffs
while still creating a record that a change occurred:

- **`AIProvider.settings`**: Encrypted JSON blob that may contain provider secrets (such as AWS Bedrock access keys and other upstream LLM credentials).
  The audit log records that settings changed, but not the new value.
- **`Chat.title`**: May contain sensitive content.
  The audit log records that a chat was created or modified, but not its title.
- **`UserSecret.value`**: The secret value.
  The audit log records create, write, and delete events but not the actual secret material.

## Querying audit log events

### UI

In the Coder dashboard, navigate to **Deployment** > **Audit Logs**.
Use the **Resource type** filter to narrow results:

- `ai_provider` - AI provider configuration and key changes
- `ai_seat` - First AI seat consumption per user
- `auditable_group_ai_budget` - Group AI spend budget changes
- `chat` - Chat session creation and modification
- `task` - Coder Task creation, updates, and deletion
- `user_secret` - Personal secret creation, updates, and deletion

### API

Use the [`GET /api/v2/audit`](../reference/api/audit.md#get-audit-logs) endpoint
with the `resource_type` query parameter:

```bash
# List all AI provider configuration and key changes
curl "$CODER_URL/api/v2/audit?resource_type=ai_provider" \
     -H "Coder-Session-Token: $CODER_SESSION_TOKEN" | jq .

# List first AI seat activations (one event per user)
curl "$CODER_URL/api/v2/audit?resource_type=ai_seat" \
     -H "Coder-Session-Token: $CODER_SESSION_TOKEN" | jq .

# List group AI budget changes
curl "$CODER_URL/api/v2/audit?resource_type=auditable_group_ai_budget" \
     -H "Coder-Session-Token: $CODER_SESSION_TOKEN" | jq .

# List chat events for a specific user
curl "$CODER_URL/api/v2/audit?resource_type=chat&username=alice" \
     -H "Coder-Session-Token: $CODER_SESSION_TOKEN" | jq .
```

## AI Gateway session audit

AI Gateway session data, including prompts, token usage, tool calls, and model reasoning,
is stored separately from the `audit_logs` table in the `aibridge` session store.
It is not accessible through the Audit Logs UI or API.

See [Auditing AI Sessions](./ai-gateway/audit.md) for details on navigating session history,
tracing agentic loops, and conducting forensic audits of AI activity.

## Agent Firewall events

Agent Firewall enforcement decisions (allow or deny) are emitted to workspace logs and the connection log.
These are not written to the `audit_logs` table.

To review Agent Firewall activity:

- **Workspace logs**: Navigate to the workspace and open the agent log tab.
- **Connection log API**: Use [`GET /api/v2/connectionlog`](../reference/api/enterprise.md#get-connection-logs) to export structured connection events.

See [Agent Firewall](./agent-firewall/index.md) for configuration and policy guidance.
