# Feature comparison

This page compares Coder's two approaches for running coding agents with
four widely used agent products: Anthropic's Claude Code, OpenAI's Codex,
Cursor, and Devin (Cognition). It is written to be factual rather than
promotional: cells link to the vendor documentation that supports them, and
the [gaps section](#where-coder-agents-falls-short-today) lists areas where
Coder Agents currently trails the market.

All external sources were verified on June 12, 2026. These products ship
weekly; check the linked vendor documentation for the current state.

## How to read this page

The matrix has two Coder columns:

- **Coder Agents** is the native, self-hosted harness built into the
  open-source `coder/coder` control plane. The agent loop runs in your
  deployment and executes tools in your workspaces. See
  [Architecture](./architecture.md).
- **Coder Tasks + BYO agent** runs a third-party terminal agent (Claude
  Code, Codex CLI, Goose, Aider, and others) inside a Coder workspace via
  [Coder Tasks](../tasks.md). Capabilities marked "agent-dependent" come
  from whichever harness you bring, not from Coder.

> [!NOTE]
> Coder Tasks moves to a 12-month Extended Support Release for Premium
> customers starting June 2, 2026, and is removed from new Coder releases
> beginning with v2.37. [Coder Agents](./index.md) is the long-term
> replacement. See the [Tasks documentation](../tasks.md) for details.

For the other columns, products span multiple surfaces (CLI, IDE, cloud)
that behave differently. Cells call out the variant when it matters, for
example "CLI" versus "cloud".

## Where Coder Agents falls short today

Coder Agents is in Beta, and this comparison surfaces real gaps. Listing
them here keeps the page honest and doubles as roadmap input:

- **No lifecycle hooks.** Claude Code, Codex, and Cursor all expose
  pre/post tool-call hooks, and Claude Code's event model is becoming a
  de-facto standard that Cursor and the Devin CLI advertise compatibility
  with. Coder Agents has no equivalent.
- **No custom slash commands, plugins, or skill distribution.** Skills
  exist (workspace and personal), but there is no plugin system,
  marketplace, or org-level distribution mechanism.
- **No per-tool-call approvals or command allowlists.** Plan mode is the
  only approval gate. Competitors pair approval prompts with OS-level
  command sandboxing; Coder Agents relies on workspace isolation and
  template-level network policy instead.
- **Web and API only.** There is no IDE extension, desktop or mobile app,
  or full-featured chat CLI for Coder Agents.
- **No built-in chat or issue-tracker integrations.** Slack, Teams, Jira,
  and Linear integrations are do-it-yourself via the Chats API, while all
  four competitors ship at least Slack natively.
- **No dedicated code review product.** Competitors ship purpose-built
  reviewers (Claude Code Review, Codex GitHub review, Cursor Bugbot, Devin
  Review). Coder Agents offers a diff viewer and skill-based review
  workflows.
- **No file checkpoints or rewind UI.** Editing a message rewinds the
  conversation, but there is no file-state checkpoint and restore like
  Claude Code or Cursor.
- **Browser and computer use is experimental** and requires an opt-in
  experiment, while competitors have shipped browser control to general
  availability.
- **No codebase indexing or semantic search.** Exploration is tool-driven
  (shell, grep) only.
- **No scheduled or event-triggered runs.** There is no equivalent of
  Claude Code Routines, Codex Automations, or Devin Automations beyond
  API-driven triggers you build yourself.
- **No OpenTelemetry export for agent sessions**, and no Google Vertex AI
  provider type (Google models are served via the Gemini API type or an
  OpenAI-compatible gateway).

## Where the architecture differs

The architectural rows above reduce to one distinction: who runs the agent
loop and who sits between your code and the model.

- With **Coder Agents**, the loop, control plane, chat history, and tool
  execution all run on infrastructure you operate, and prompts go directly
  from your control plane to the provider you configure. With a self-hosted
  model endpoint, the entire system can run inside your network.
- **Claude Code** and **Codex CLI** run the loop on the developer's
  machine, but inference requires their vendor's models (Codex CLI is the
  exception: it can target local open-source models). Their cloud surfaces
  move the loop into vendor infrastructure.
- **Cursor** executes tools locally or on self-hosted workers, but
  orchestration and all model traffic flow through Cursor's cloud, even
  with customer-supplied API keys.
- **Devin** keeps both the agent brain and (by default) execution in
  Cognition's cloud; the dedicated VPC option isolates the Devbox but is
  still Cognition-managed.

None of this makes one product universally better. Vendor-hosted loops buy
turnkey cloud parallelism and integrations that Coder Agents currently
lacks; a self-hosted loop buys data control, air-gap capability, and
auditability that vendor clouds cannot offer.
