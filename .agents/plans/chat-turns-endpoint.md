# Chat Turns Endpoint

## Goal

Add a paginated, lightweight `/api/experimental/chats/{chat}/turns`
endpoint that returns one row per "agent turn" (a user message and the
assistant work that followed it) with: tool call count, total cost,
duration. Designed so the frontend can render a collapsed timeline of
hundreds of turns and only load full message contents on expand.

A "turn" is the half-open id range `[user_msg.id, next_user_msg.id)`
within a chat, where messages are filtered to `visibility IN ('user',
'both')` and `deleted = false` (matching `GetChatMessages*`).

## Performance approach

The frontend renders the timeline frequently, including in long chats
with many turns. The query must avoid scanning JSONB content per row.

Plan:

1. **Denormalize `tool_call_count`** onto `chat_messages` as a
   `smallint NOT NULL DEFAULT 0`. Backfill with one pass over existing
   rows. Maintain on insert via `InsertChatMessages` CTE
   (computed from `content` JSONB) and on
   `UpdateChatMessageByID` (recomputed when `content` is updated).
2. The turns query uses a single window-function pass over
   `chat_messages` filtered by `chat_id`:
   - `LEAD(id) OVER (...)` gives the next user message id (turn end).
   - Then aggregate per turn: SUM(tool_call_count), SUM(total_cost_micros),
     MIN(created_at) (turn start), MAX(created_at) (turn end / last
     message).
3. Existing indexes are sufficient:
   `idx_chat_messages_chat_created (chat_id, created_at)` already
   covers the per-chat scan, and the primary key on `id` gives us the
   ascending order. We add a partial index for fast cursor lookups
   when filtering by `role = 'user'` (used to translate the
   `before_id`/`after_id` user message cursors).

## Endpoint shape

`GET /api/experimental/chats/{chat}/turns?before_id=&after_id=&limit=`

- `before_id`: return turns whose `start_message_id < before_id`
- `after_id`: return turns whose `start_message_id > after_id`
- `limit`: 1-200, default 50

Returns:

```json
{
  "turns": [
    {
      "start_message_id": 123,
      "end_message_id": 145,        // exclusive; next user msg or null
      "user_message_id": 123,
      "started_at": "...",
      "last_message_at": "...",
      "duration_ms": 18234,
      "tool_call_count": 7,
      "assistant_message_count": 4,
      "total_cost_micros": 8421,
      "is_open": false              // true if no following user msg yet
    }
  ],
  "has_more": false
}
```

Default order is DESC (newest first), matching `/messages`.

## Open considerations

- We treat `tool_call_count` as 0 for non-assistant messages. Backfill
  query also computes it for assistant rows only ,  tool-role messages
  may carry tool-result parts but we are counting **calls**, not
  results.
- For an in-flight turn (no following user message), `end_message_id`
  is `NULL` and `is_open` is `true`. The frontend can then choose to
  default-expand only the open turn while collapsing completed turns.
